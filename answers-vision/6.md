# Lecture 6 Answers

## Exercise 6.1

The system returns:

```text
8 relevant documents
10 nonrelevant documents
18 retrieved documents total
20 relevant documents total in the collection
```

Precision:

```text
P = relevant retrieved / retrieved
  = 8 / (8 + 10)
  = 8 / 18
  = 4 / 9
  ≈ 0.444
```

Recall:

```text
R = relevant retrieved / relevant in collection
  = 8 / 20
  = 0.4
```

So precision is about `44.4%`, and recall is `40%`.

## Exercise 6.2

The two formulas are:

```text
F = 1 / (alpha * (1/P) + (1 - alpha) * (1/R))
```

and

```text
F_beta = ((beta^2 + 1) * P * R) / (beta^2 * P + R)
```

Given:

```text
alpha = 1 / (beta^2 + 1)
```

we also have:

```text
1 - alpha = 1 - 1 / (beta^2 + 1)
          = beta^2 / (beta^2 + 1)
```

Substitute into the first formula:

```text
F = 1 / ( 1/((beta^2 + 1)P) + beta^2/((beta^2 + 1)R) )
```

Put the denominator over a common denominator:

```text
1/((beta^2 + 1)P) + beta^2/((beta^2 + 1)R)
= (R + beta^2 P) / ((beta^2 + 1) P R)
```

Therefore:

```text
F = ((beta^2 + 1) P R) / (beta^2 P + R)
```

which is the `F_beta` formula.

## Exercise 6.3

No, there does not have to be a break-even point where precision equals recall.

A counterexample is a ranked result list where the precision-recall points never satisfy `P = R`. Suppose there are 4 relevant documents in the collection, but the system returns only:

```text
R N N
```

The precision-recall values after each prefix are:

| rank | relevant retrieved | precision | recall |
| ---: | ---: | ---: | ---: |
| 1 | 1 | 1.000 | 0.250 |
| 2 | 1 | 0.500 | 0.250 |
| 3 | 1 | 0.333 | 0.250 |

There is no prefix with `precision = recall`. Thus a break-even point is not guaranteed.

## Exercise 6.4

At a break-even point, precision and recall are equal:

```text
P = R = b
```

The balanced F measure is:

```text
F1 = 2PR / (P + R)
```

Substitute `P = R = b`:

```text
F1 = 2b^2 / (b + b)
   = 2b^2 / 2b
   = b
```

So if a system has a break-even point at value `b`, then `F1` at that point is also `b`.

This does not mean the maximum `F1` over the ranked list must equal the break-even point. It only says that when `P = R`, the harmonic mean is the same value.

## Exercise 6.5

The ranked list is:

```text
R R N N N N N N R N R N N N R N N N N R
```

The relevant documents occur at ranks:

```text
1, 2, 9, 11, 15, 20
```

There are 8 relevant documents total in the collection.

### Precision at top 20

There are 6 relevant documents in the top 20:

```text
P@20 = 6 / 20 = 0.3
```

So precision at top 20 is `30%`.

### F1 at top 20

Recall at top 20 is:

```text
R@20 = 6 / 8 = 0.75
```

Therefore:

```text
F1 = 2PR / (P + R)
   = 2 * 0.3 * 0.75 / (0.3 + 0.75)
   = 0.45 / 1.05
   ≈ 0.4286
```

### Uninterpolated precision at 25% recall

Since there are 8 relevant documents total, 25% recall means retrieving:

```text
0.25 * 8 = 2 relevant documents
```

The second relevant document is at rank 2:

```text
P@2 = 2 / 2 = 1.0
```

So the uninterpolated precision at 25% recall is `1.0`.

### Interpolated precision at 33% recall

The precision-recall points at relevant ranks are:

| relevant rank | rank | precision | recall |
| ---: | ---: | ---: | ---: |
| 1 | 1 | 1/1 = 1.0000 | 1/8 = 0.125 |
| 2 | 2 | 2/2 = 1.0000 | 2/8 = 0.250 |
| 3 | 9 | 3/9 = 0.3333 | 3/8 = 0.375 |
| 4 | 11 | 4/11 = 0.3636 | 4/8 = 0.500 |
| 5 | 15 | 5/15 = 0.3333 | 5/8 = 0.625 |
| 6 | 20 | 6/20 = 0.3000 | 6/8 = 0.750 |

Interpolated precision at recall `0.33` is the maximum precision at any recall level at least `0.33`.

The future points have precisions:

```text
0.3333, 0.3636, 0.3333, 0.3000
```

The maximum is:

```text
4/11 ≈ 0.3636
```

So interpolated precision at 33% recall is about `0.364`.

### MAP for the query

For a single query, MAP is the same as average precision. Average precision is the mean of precision values at the ranks where relevant documents are retrieved, divided by the total number of relevant documents.

```text
AP = (P@1 + P@2 + P@9 + P@11 + P@15 + P@20) / 8
   = (1 + 1 + 3/9 + 4/11 + 5/15 + 6/20) / 8
   ≈ 3.3303 / 8
   ≈ 0.4163
```

The denominator is 8 because there are 8 relevant documents in the collection. The two relevant documents not returned in the complete result set contribute 0.

So the MAP for this single query is approximately `0.416`.
