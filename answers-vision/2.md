# Lecture 2 Answers

## Exercise 2.1

An IR system can combine stop words and a positional index by indexing normal terms positionally and also keeping enough information about stop words to answer phrase and proximity queries.

A practical design is:

1. Use a normal inverted index for ordinary term lookup.
2. Store positions in postings, so phrase and proximity queries can be checked.
3. Either keep stop words in the positional index, or remove them from the main index but preserve original token positions and verify phrases containing stop words by postfiltering candidate documents against the original text.

The problem is that stop words are extremely frequent. Their postings lists are huge, and positional postings for them are even larger because every occurrence needs a position. Phrase queries such as `"king of denmark"` need the stop word `of`, but indexing all stop words makes the index bigger and can make positional intersection expensive.

If stop words are not indexed, then phrase queries with stop words can become imprecise. For example, removing `of` and searching for `king` followed two positions later by `denmark` may match any word between them, not necessarily `of`. This can be handled by keeping stop words in a separate positional structure, by retaining stop words in the positional index despite the cost, or by using the positional index only to generate candidates and then checking the exact phrase in the original document text.

## Exercise 2.2

In the standard permuterm index, an exact permuterm dictionary entry can come from only one original vocabulary term.

Reason: each original term is first extended with the special boundary symbol `$`, and rotations are stored. Since `$` occurs exactly once and is not a normal character, any exact rotation can be rotated back uniquely to the original term.

For example, `hello` generates rotations of `hello$`. A dictionary entry such as `lo$hel` uniquely reconstructs `hello`.

Wildcard lookup can return many original terms because we do a prefix search over permuterm keys, such as `o$hel*`, but each exact permuterm key maps to one original vocabulary term.

## Exercise 2.3

For the term `mama`, append the end marker `$`:

```text
mama$
```

The permuterm dictionary entries are the rotations:

```text
mama$ -> mama
ama$m -> mama
ma$ma -> mama
a$mam -> mama
$mama -> mama
```

The `$mama` rotation is the one needed for prefix wildcard queries such as `mam*`. Some slide examples omit this rotation when listing examples, but the standard permuterm index includes it.

## Exercise 2.4

For a wildcard query of the form:

```text
s*ng
```

use the permuterm rule for `X*Y`: look up `Y$X*`.

Here `X = s` and `Y = ng`, so the lookup key is:

```text
ng$s*
```

This is a prefix lookup in the permuterm dictionary. It finds permuterm entries such as `ng$si...`, which correspond to original terms like `sing`.

## Exercise 2.5

Lexicographic ordering is useful because wildcard processing with a k-gram index requires intersecting postings lists of vocabulary terms.

For example, a query may be converted into several k-grams. Each k-gram points to the vocabulary terms containing it. If those term lists are sorted lexicographically, the system can intersect them efficiently using the same kind of linear merge used for sorted postings lists in an inverted index.

The ordering also makes it easy to remove duplicates, produce deterministic candidate lists, and then postfilter the surviving terms against the original wildcard pattern.
