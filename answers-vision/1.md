## Exercise 1.1

Using lowercased terms and document-only postings:

| term | postings |
| --- | --- |
| rise | Doc2, Doc4 |
| home | Doc1, Doc2, Doc3, Doc4 |

If the index is positional, with token positions counted from 1:

| term | positional postings |
| --- | --- |
| rise | Doc2: 3; Doc4: 5 |
| home | Doc1: 2; Doc2: 1; Doc3: 3; Doc4: 3 |

The document-only postings record that a term occurs in a document. The positional postings additionally record where the term occurs, which is needed for phrase and proximity queries.

## Exercise 1.2

For `trees AND skies AND kaleidoscope`, process the conjuncts in increasing postings-list size:

1. `kaleidoscope` - 87,009
2. `skies` - 271,658
3. `trees` - 316,812

So a good order is:

```text
(kaleidoscope AND skies) AND trees
```

For `(tangerine OR trees) AND (marmalade OR skies) AND (kaleidoscope OR eyes)`, first estimate the size of each disjunctive clause by adding the postings sizes. This gives an upper bound because the two lists may overlap:

| clause | estimated size |
| --- | ---: |
| `kaleidoscope OR eyes` | 87,009 + 213,312 = 300,321 |
| `tangerine OR trees` | 46,653 + 316,812 = 363,465 |
| `marmalade OR skies` | 107,913 + 271,658 = 379,571 |

Then process the resulting conjuncts from smallest to largest:

```text
(kaleidoscope OR eyes)
AND (tangerine OR trees)
AND (marmalade OR skies)
```

Within each `OR`, the two postings lists are merged to form the union. The main saving comes from doing the smallest conjunctive intermediate result first.

## Exercise 1.3

No. Processing postings lists in increasing size is a very good heuristic, but it is not guaranteed to be optimal because the sizes of intermediate intersections depend on how the postings lists overlap.

Example:

```text
A = {1, 2, ..., 99}        size 99
B = {1, 2, ..., 100}       size 100
C = {101, 102, ..., 200}   size 100
```

For the query `A AND B AND C`, increasing-size order starts with `A AND B`, producing `{1, 2, ..., 99}`, and then intersects that with `C`, producing the empty set. But `B AND C` is already empty, so starting with `B AND C` can terminate the query earlier. The smaller-first order did not exploit the fact that the two larger postings lists were disjoint.

Thus postings-list size alone is not enough information for a guaranteed optimal order; overlap and early termination also matter.

## Exercise 1.4

For `x OR y`, return the union of the two sorted postings lists. The merge is similar to intersection, but we output every document that occurs in either list and avoid duplicates.

```text
Union(p1, p2)
    answer <- <>

    while p1 != nil and p2 != nil
        if docID(p1) = docID(p2)
            Add(answer, docID(p1))
            p1 <- next(p1)
            p2 <- next(p2)
        else if docID(p1) < docID(p2)
            Add(answer, docID(p1))
            p1 <- next(p1)
        else
            Add(answer, docID(p2))
            p2 <- next(p2)

    while p1 != nil
        Add(answer, docID(p1))
        p1 <- next(p1)

    while p2 != nil
        Add(answer, docID(p2))
        p2 <- next(p2)

    return answer
```

This is linear in the lengths of the two postings lists.

## Exercise 1.5

The query `x AND NOT y` should be handled as a set difference:

```text
postings(x) - postings(y)
```

Naive evaluation is expensive because `NOT y` means all documents in the collection that do not contain `y`. Materializing this complement can be enormous: if the collection has `N` documents and `y` occurs in only a small number of them, then `NOT y` is almost the whole collection. Intersecting `x` with that huge list wastes work.

Instead, scan `x` and `y` together and output only documents from `x` that are not in `y`:

```text
Difference(p1, p2)    // p1 = postings(x), p2 = postings(y)
    answer <- <>

    while p1 != nil and p2 != nil
        if docID(p1) = docID(p2)
            p1 <- next(p1)
            p2 <- next(p2)
        else if docID(p1) < docID(p2)
            Add(answer, docID(p1))
            p1 <- next(p1)
        else
            p2 <- next(p2)

    while p1 != nil
        Add(answer, docID(p1))
        p1 <- next(p1)

    return answer
```

This avoids constructing the complement of `y`. The output can never be larger than `postings(x)`.

## Exercise 1.6

| statement | true/false | explanation |
| --- | --- | --- |
| In a Boolean retrieval system, stemming never lowers precision. | False | Stemming can conflate distinct meanings, causing false positives. For example, conflating `universe` and `university` would retrieve documents about one when the user wanted the other. |
| In a Boolean retrieval system, stemming never lowers recall. | False as an absolute statement | For simple positive term queries, consistent stemming normally preserves or increases recall because more variants match. But Boolean queries may include operators such as `NOT`; after stemming, a negative term can exclude more documents than intended. So "never" is too strong. |
| Stemming increases the size of the vocabulary. | False | Stemming maps multiple surface forms to one stem, so it normally reduces the vocabulary size. |
| Stemming should be invoked at indexing time but not while processing a query. | False | Stemming must be applied consistently at indexing time and query time. Otherwise query terms may not match the stemmed terms stored in the dictionary. |

## Exercise 1.7

Pairs I would not conflate:

| pair | should they be conflated? | reasoning |
| --- | --- | --- |
| `abandon` / `abandonment` | Usually acceptable | These are closely related verb/noun forms. A user interested in one will often accept documents containing the other. |
| `absorbency` / `absorbent` | Debatable, often acceptable | `absorbency` is the property and `absorbent` is the adjective or noun for something having that property. They are not identical, but in many IR settings the semantic relation is close enough. |
| `marketing` / `markets` | Often should not be conflated | `marketing` means the activity or field, while `markets` can mean places, institutions, or plural instances of `market`. Some queries about financial markets should not be expanded to marketing. |
| `university` / `universe` | Should not be conflated | These have unrelated meanings despite similar spelling. Conflating them would significantly hurt precision. |
| `volume` / `volumes` | Should be conflated | This is a regular singular/plural inflection and is normally exactly the kind of variation stemming should handle. |

The most clearly wrong conflation is `university` / `universe`. The pair `marketing` / `markets` is also risky because the meanings can diverge substantially depending on the query.
