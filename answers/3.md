# Lecture 3 Answers

## Key Concepts

**Sort-based index construction (BSBI):** builds an index by collecting (termID, docID) pairs in memory, sorting them when a block is full, writing to disk, and merging all sorted blocks at the end. Requires knowing all termIDs in advance.

**Blocked sort-based indexing (BSBI):** the standard sort-based approach that processes the collection in fixed-size blocks that fit in memory, sorts each block, writes it to disk, then merges the blocks.

**Single-pass in-memory indexing (SPIMI):** builds an inverted index in one pass without sorting, by directly appending to postings lists in a hash table. When memory is full, the partial index is flushed to disk. More memory-efficient than BSBI because it avoids storing termID pairs separately.

**Distributed indexing:** splitting the index construction task across many machines. The collection is divided into document partitions (or term partitions) and processed in parallel.

**MapReduce:** a programming model for distributed computation. The *map* phase emits (key, value) pairs (e.g. (term, docID)); the *reduce* phase aggregates all values for each key (e.g. builds the postings list for each term).

**Dynamic indexing:** maintaining an index that stays up to date as new documents are added or deleted, without rebuilding from scratch. Uses an auxiliary in-memory index for new documents that is periodically merged into the main index.

**Logarithmic index merging:** a dynamic indexing strategy that maintains a series of disk indexes of exponentially increasing sizes (I₀, I₁, I₂, ...). When an in-memory index fills up it is merged into the smallest disk index, cascading merges up as needed. Reduces the total number of merge operations compared with merging into a single index each time.

## Exercise 3.1

> Given the statistics of the Reuters collection on slide 38.
>
> 1. What is the average document frequency of a term?
> 2. Explain the difference of 4.5 bytes per token vs. 7.5 bytes per type.
> 3. How many positional postings?

From the Reuters statistics:

```text
N = 800,000 documents
L = 200 tokens per document on average
M = 400,000 terms / word types
non-positional postings = 100,000,000
```

The average document frequency of a term is:

```text
100,000,000 / 400,000 = 250
```

So an average term occurs in about 250 documents.

The difference between `4.5 bytes per token` and `7.5 bytes per type` is that tokens and types are averaged over different distributions:

- A token is an occurrence in running text. Very frequent words tend to be short, such as `the`, `of`, `and`, so the token-weighted average is lower.
- A type is a distinct vocabulary term. Each distinct word is counted once, so rare long words count just as much as frequent short words. This raises the average length.

The positional postings count is approximately the number of token occurrences:

```text
N * L = 800,000 * 200 = 160,000,000
```

So the model statistics imply about 160 million positional postings. The later exact preprocessing table in the slides gives `197,879,290` unfiltered positional postings; the difference is because `L = 200` is a rounded model average.

## Exercise 3.2

> For n = 2 and 1 ≤ T ≤ 30, perform a step-by-step simulation of the Logarithmic merge algorithm. Create a table that shows, for each point in time at which T = 2∗k tokens have been processed (1 ≤ k ≤ 15), which of the three indexes l0, ..., l3 are in use. The first three lines of the table are given below.
>
> | T | I3 | I2 | I1 | I0 |
> | ---: | ---: | ---: | ---: | ---: |
> | 2 | 0 | 0 | 0 | 0 |
> | 4 | 0 | 0 | 0 | 1 |
> | 6 | 0 | 0 | 1 | 0 |
> | 8 | ? | ? | ? | ? |

For `n = 2`, each disk index `Ii` stores a block of size `2 * 2^i` tokens. The first three rows given in the question imply that, at time `T = 2k`, the current two-token in-memory index `Z0` has not yet been written to disk. Therefore the disk indexes represent `k - 1` flushed blocks of size 2, written in binary across `I0, I1, I2, ...`.

| T | I3 | I2 | I1 | I0 |
| ---: | ---: | ---: | ---: | ---: |
| 2 | 0 | 0 | 0 | 0 |
| 4 | 0 | 0 | 0 | 1 |
| 6 | 0 | 0 | 1 | 0 |
| 8 | 0 | 0 | 1 | 1 |
| 10 | 0 | 1 | 0 | 0 |
| 12 | 0 | 1 | 0 | 1 |
| 14 | 0 | 1 | 1 | 0 |
| 16 | 0 | 1 | 1 | 1 |
| 18 | 1 | 0 | 0 | 0 |
| 20 | 1 | 0 | 0 | 1 |
| 22 | 1 | 0 | 1 | 0 |
| 24 | 1 | 0 | 1 | 1 |
| 26 | 1 | 1 | 0 | 0 |
| 28 | 1 | 1 | 0 | 1 |
| 30 | 1 | 1 | 1 | 0 |

For example, at `T = 14`, we have `k = 7`, so `k - 1 = 6`, whose binary representation is `0110`; therefore `I2` and `I1` are present, while `I3` and `I0` are absent.

## Exercise 3.3

> An auxiliary index can impair the quality of collection statistics. An example is the term weighting method idf, which is defined as log(N/df_i) where N is the total number of documents and df_i is the number of documents that term i occurs in. Show that even a small auxiliary index can cause significant error in idf when it is computed on the main index only. Consider a rare term that suddenly occurs frequently (e.g., Flossie as in Tropical Storm Flossie).

If idf is computed only on the main index, then a term that was rare in the old collection but frequent in the auxiliary index can receive a much too large idf.

Let:

```text
N_main = 1,000,000
df_main(Flossie) = 1
N_aux = 1,000
df_aux(Flossie) = 1,000
```

The auxiliary index is small: only `1,000 / 1,000,000 = 0.1%` of the main collection size. But suppose every new auxiliary document talks about Tropical Storm Flossie.

If idf is computed from the main index only:

```text
idf_main = log(1,000,000 / 1) = log(1,000,000)
```

The correct idf over the combined collection is:

```text
idf_true = log((1,000,000 + 1,000) / (1 + 1,000))
         = log(1,001,000 / 1,001)
         = log(1,000)
```

Using natural logarithms:

```text
idf_main ≈ 13.82
idf_true ≈ 6.91
```

So the main-index-only idf is about twice as large as it should be, even though the auxiliary index is only 0.1% of the collection. The problem is especially severe for rare terms because a small absolute number of new documents can drastically change their document frequency.
