## Key Concepts

**Index compression:** reducing the size of the dictionary and postings on disk to save space and speed up I/O. Exploits the statistical structure of term frequencies and docID gaps.

**Lossy vs. lossless compression:** lossless compression (e.g. gap encoding, variable-byte codes) allows perfect reconstruction of the original data. Lossy compression (e.g. stopping, case folding, stemming) discards some information and cannot be reversed.

**Heap's law:** an empirical law stating that the vocabulary size M grows as M â‰ˆ kT^b where T is the number of tokens, k â‰ˆ 30â€“100, and b â‰ˆ 0.5. Vocabulary keeps growing as the collection grows, just more slowly.

**Zipf's law:** an empirical law stating that the frequency of the r-th most frequent term is proportional to 1/r. A few terms are very frequent; most terms are rare. This motivates both stopword removal and compression schemes.

**Dictionary compression as a string:** instead of fixed-length entries, concatenate all terms into one long string and store only a pointer, document frequency, and term frequency per entry. Saves space wasted on padding.

**Blocking:** a dictionary compression technique that groups k consecutive terms and stores only one pointer per block plus a length byte per term, saving (kâˆ’1) pointers per block at the cost of linear scan within a block.

**Front coding:** compresses the dictionary by storing only the characters that differ from the previous term, since the sorted dictionary has many common prefixes (e.g. `automate`, `automatic`, `automation` â†’ store the common prefix once).

**Gap encoding (postings compression):** instead of storing absolute docIDs in postings lists, store the differences (gaps) between consecutive docIDs. Gaps are small for common terms, enabling short variable-length codes.

**Isolated spelling correction:** correcting a misspelled word without considering its context â€” treats each query term independently. Uses edit distance to find the closest vocabulary term.

**Context-sensitive spelling correction:** uses surrounding words to choose the best correction, catching errors like `their` vs. `there` where the misspelled form is itself a valid word.

**Spelling correction in IR:** applied at query time to suggest or auto-correct query terms that do not appear in the index. Can use the index vocabulary itself as the dictionary of valid terms.

**(Weighted) edit distance:** the minimum number of weighted edit operations (insert, delete, substitute) needed to transform one string into another. Weights allow making some errors cheaper than others (e.g. nearby keyboard keys).

**Levenshtein distance:** the standard unweighted edit distance where each insertion, deletion, and substitution costs 1. Computed in O(mn) time via dynamic programming.

> Assuming one machine word per posting, what is the size of the uncompressed (nonpositional) index for different tokenizations based on Slide 48? How do these numbers compare with numbers on Slide 71?

Assume one machine word per nonpositional posting, and take one machine word to be 32 bits = 4 bytes. Using the Reuters preprocessing table:

| tokenization / preprocessing | nonpositional postings | uncompressed postings size |
| --- | ---: | ---: |
| unfiltered | 109,971,179 | 439.9 MB |
| no numbers | 100,680,242 | 402.7 MB |

| 30 stop words removed | 83,390,443 | 333.6 MB |
| 150 stop words removed | 67,001,847 | 268.0 MB |
| stemming | 63,812,300 | 255.2 MB |

These numbers are just:

```text
number of postings * 4 bytes
```

Compared with the Reuters compression summary, the slide value `postings, uncompressed (32-bit words) = 400 MB` corresponds to roughly 100 million postings stored in 4-byte words. The `20 bits` estimate is `100,000,000 * 20 / 8 = 250 MB`, which is smaller because Reuters has 800,000 documents and therefore needs only about `log2(800,000) < 20` bits per docID. Variable-byte encoding reduces this further to about `116 MB` by using gap encoding and shorter codes for small gaps.

## Exercise 4.2

> Estimate the space usage of the Reuters-RCV1 dictionary with blocks of size k = 8 and k = 16 in blocked dictionary storage.

Without blocking, dictionary-as-a-string storage uses:

```text
400,000 * (4 + 4 + 3 + 8) = 7.6 MB
```

The `3` is the term pointer size. With blocking of size `k`, we replace `k` separate 3-byte term pointers with:

```text
3 bytes for one block pointer + k bytes for term lengths
```

So the saving per block is:

```text
3k - (3 + k) = 2k - 3 bytes
```

For `k = 8`:

```text
number of blocks = 400,000 / 8 = 50,000
saving = 50,000 * (2*8 - 3)
       = 50,000 * 13
       = 650,000 bytes = 0.65 MB
size â‰ˆ 7.6 - 0.65 = 6.95 MB
```

For `k = 16`:

```text
number of blocks = 400,000 / 16 = 25,000
saving = 25,000 * (2*16 - 3)
       = 25,000 * 29
       = 725,000 bytes = 0.725 MB
size â‰ˆ 7.6 - 0.725 = 6.875 MB
```

So the estimates are about `7.0 MB` for `k = 8` and `6.9 MB` for `k = 16`. Larger blocks save a little more space, but lookup is slower because more terms inside a block may need to be scanned.

## Exercise 4.3

> If |a| denotes the length of string a, show that the edit distance between a and b is never more than max{|a|, |b|}.

Let `|a| = m` and `|b| = n`.

If `m <= n`, transform `a` into `b` as follows:

1. Replace the first `m` characters of `a` by the first `m` characters of `b`, costing at most `m`.
2. Insert the remaining `n - m` characters of `b`, costing `n - m`.

Total cost:

```text
m + (n - m) = n = max{m, n}
```

If `m > n`, do the symmetric operation:

1. Replace the first `n` characters of `a` by the `n` characters of `b`, costing at most `n`.
2. Delete the remaining `m - n` characters of `a`, costing `m - n`.

Total cost:

```text
n + (m - n) = m = max{m, n}
```

Since this gives an explicit edit sequence with cost at most `max{|a|, |b|}`, the minimum possible edit distance can never be larger than that.

## Exercise 4.5

> Consider the four-term query *catched in the rye* and suppose that each of the query terms has five alternative terms suggested by isolated-term correction. How many possible corrected phrases must we consider if we do not trim the space of corrected phrases, but instead try all six variants for each of the terms?

Each of the 4 query terms has:

```text
5 suggested alternatives + 1 original term = 6 variants
```

Trying all combinations gives:

```text
6^4 = 1296
```

So we must consider `1296` possible corrected phrases if we do not trim the space.
